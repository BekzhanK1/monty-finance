# Food Service: архитектура и roadmap к superapp

Документ для обсуждения: меню «что готовим дома», данные, логика и связка с остальным Monty как задел под **superapp** (несколько вертикалей в одном продукте).

### Модель «пара = один актёр»

Продуктом пользуетесь **вы двое как одна комната/один дом**: не нужны роли, раздельные «мои»/«твои» меню или права на уровне Food. В БД и API достаточно **одного рабочего пространства** (тот же `household`/tenant, что и в Finance, или один `user_id`, если так проще в текущей схеме): все рецепты, слоты меню и списки покупок общие. Два Telegram-аккаунта могут ходить в одно и то же — это просто два клиента к одним данным; конфликты редки и на MVP хватает **last-write-wins**.

---

## 1. Позиционирование в superapp

| Слой | Идея |
|------|------|
| **Identity** | Один общий контекст для пары: одно меню, один список покупок, как один логический «человек» с точки зрения домена Food. |
| **Вертикали** | Finance, Food, позже Health, Travel, … — общий shell (навигация уже через «Все сервисы»). |
| **Общие сервисы** | Auth (JWT / Telegram), уведомления, возможно общий «семейный» контекст в БД. |
| **Интеграции** | Telegram: напоминания, быстрый ввод «сегодня суп»; в перспективе — связка с бюджетом (оценка стоимости блюда). |

Практичный старт: **Food живёт в том же `monty-backend`**, пакет [`app/food/`](../monty-backend/app/food/) (роутер `/food/...`, модели `food_*`). На фронте — [`monty-frontend/src/food/`](../monty-frontend/src/food/) и [`src/services/food.ts`](../monty-frontend/src/services/food.ts). Позже при росте можно вынести в микросервис без смены контракта на фронте (BFF или gateway).

## 2. Домен и бизнес-логика (MVP → расширение)

### MVP (реально закрыть «меню дома»)

1. **Недельное меню** — привязка к календарной неделе и дню: завтрак/обед/ужин (опционально слоты).
2. **Блюда** — название, заметки, ссылка на рецепт (URL или текст), теги (быстро, веган, batch).
3. **Список покупок** — агрегат ингредиентов из запланированных блюд за период + ручные позиции; галочки «куплено».

### Следующий слой

4. **Рецепты с ингредиентами** — нормализованные строки (продукт + количество + единица), чтобы умножать на порции и не дублировать «лук» в списке.
5. **Кладовая (pantry)** — что уже есть дома; при генерации списка вычитать или помечать «докупить мало».
6. **Повтор и шаблоны** — «как прошлую неделю», избранные недели, типовые наборы (например «рабочая неделя»).

### «Прикольные» фичи (superapp-уровень)

| Фича | Зачем |
|------|--------|
| **AI-предложение меню** | По ограничениям (время, бюджет, диета), остаткам из кладовой, сезону. У вас уже есть OpenAI в бэкенде — можно аккуратно подключить как опцию. |
| **Связка с Finance** | Оценка стоимости недели по категориям продуктов или ручным ценам; позже — импорт чеков в «продуктовый» бюджет. |
| **Голос / фото холодильника** | Быстрый ввод в Telegram: «что есть» → подсказка блюд или дополнение кладовой. |
| **Prep & batch** | «Готовим раз в воскресенье» — план заготовок и что разогреть в какой день. |
| **Два телефона, одни данные** | Realtime не обязателен; refresh или короткий polling; при желании позже WebSocket. |
| **Напоминания** | «Завтра разморозить» / «сегодня вечером X» через Telegram + push в Mini App. |
| **Локаль и привычки** | Русские названия продуктов, единицы (г, мл, шт), типовые наборы для Казахстана/РФ и т.д. |

Приоритизируй: **меню + список покупок + рецепты с ингредиентами** дают 80% ценности; остальное — дифференциация superapp.

## 3. Модель данных (черновик схемы)

Упрощённо: **одна сущность владельца данных** на пару (`household_id` или единый `owner_id` — как удобнее стыковать с текущим auth в Monty).

```text
households / workspace (один на приложение для вас двоих)
  id, ...

food_recipes
  id, household_id, title, notes, source_url, servings_default, created_at

food_recipe_ingredients
  id, recipe_id, name_raw, quantity, unit, optional_sort_order
  -- позже: product_id FK на нормализованный справочник продуктов

food_menu_weeks (опционально; можно обойтись только датами)
  id, household_id, week_start_date  -- понедельник ISO

food_menu_slots
  id, household_id, date, slot  -- enum: breakfast | lunch | dinner | snack
  recipe_id nullable, custom_title nullable  -- «пицца с друзьями» без рецепта

food_shopping_lists
  id, household_id, title, period_start, period_end, status

food_shopping_items
  id, list_id, label, quantity, unit, checked, source  -- manual | from_recipe

food_pantry_items (фаза 2)
  id, household_id, product_key or name, quantity, unit, expires_on
```

**Индексы:** `(household_id, date)` на слотах меню; `(list_id, checked)` для списка.

**Миграции:** Alembic в `monty-backend`, как для остальных таблиц.

## 4. API (черновик)

REST под префиксом `/food` (или `/api/v1/food`):

- `GET/POST /food/menu?from=&to=` — диапазон дат.
- `PUT /food/menu/slots/{id}` — назначить блюдо/текст.
- `GET/POST /food/recipes`, `GET/PATCH/DELETE /food/recipes/{id}`.
- `GET/POST /food/shopping-lists`, `POST /food/shopping-lists/{id}/generate-from-menu` — тело: `{ from, to }`.
- `PATCH /food/shopping-items/{id}` — toggle checked, количество.

Авторизация — та же, что для Finance (JWT); доступ к Food — у любого залогиненного участника **одного** общего пространства (без ветвления по «кто из пары»).

## 5. Фронтенд (Food)

- Страница **Food**: недельный вид (календарь / сетка), карточка дня, модалка выбора рецепта или свободного текста.
- Отдельный раздел **Рецепты** и **Список на неделю** (или один экран с вкладками).
- Позже: offline-first не обязателен; для superapp важнее **единый визуальный язык** с Mantine и текущим Layout.

---

## 6. Нефункциональное

- **Идемпотентность** генерации списка: либо «заменить черновик», либо версии списков.
- **Одновременное редактирование** с двух устройств: на MVP — last-write-wins; при росте можно уточнять по сущности (слот меню, пункт списка).
- **Приватность** AI: не слать персональные данные без явного согласия; минимизировать PII в промптах.

---

## 7. Предлагаемый порядок внедрения

1. Схема БД + Alembic + CRUD рецептов (минимум).
2. Слоты меню по датам + UI недели.
3. Генерация списка покупок из выбранных рецептов (даже без умного дедупа — потом улучшить).
4. Кладовая и дедуп ингредиентов.
5. Telegram-напоминания и/или AI-меню как «вау»-слой superapp.

## 8. Открытые вопросы (зафиксировать решения)

- **Закрыто для вашего кейса:** несколько household’ов не планируем — **один дом на приложение.**
- Нужны ли **порции** и масштабирование рецепта в MVP?
- Как технически маппить auth на данные Food: один `household_id` vs один дефолтный owner — переиспользовать то, что уже есть в backend.
